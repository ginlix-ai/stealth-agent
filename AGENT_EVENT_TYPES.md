# Agent Event Types Reference

This document lists all event types generated by the PTC Agent server and sent via Server-Sent Events (SSE).

## 📋 Complete List of Event Types

The agent generates **12 different event types** during execution:

### 1. **`message_chunk`** 
**Purpose**: Agent's text responses (main content)

**When Emitted**: 
- When agent generates text content
- Streamed incrementally as agent thinks/responds

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "agent": "main",
  "id": "msg-123",
  "role": "assistant",
  "content": "Hello, I can help you...",
  "content_type": "text" | "reasoning" | "reasoning_signal",
  "finish_reason": null | "stop" | "length" | "tool_calls"
}
```

**Content Types**:
- `"text"`: Regular agent response text
- `"reasoning"`: LLM internal reasoning (thinking process)
- `"reasoning_signal"`: Lifecycle signals (`"start"`, `"complete"`)

**Code Location**: `src/server/handlers/streaming_handler.py:1151`

---

### 2. **`tool_calls`**
**Purpose**: Indicates agent is using tools

**When Emitted**: 
- When agent decides to call one or more tools
- Contains complete tool name and arguments

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "agent": "main",
  "tool_calls": [
    {
      "id": "call-123",
      "name": "read_file",
      "args": {"path": "file.py"}
    }
  ]
}
```

**Code Location**: `src/server/handlers/streaming_handler.py:1078, 1134`

---

### 3. **`tool_call_chunks`**
**Purpose**: Streaming tool arguments (for animation/UI feedback)

**When Emitted**: 
- While tool arguments are being streamed
- Used for visual feedback, not for processing

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "agent": "main",
  "tool_call_id": "call-123",
  "name": "read_file",
  "chunk": "path"
}
```

**Note**: Tool arguments are already complete in `tool_calls` event. This is for UI animation only.

**Code Location**: `src/server/handlers/streaming_handler.py:856, 911, 1140`

---

### 4. **`tool_call_result`**
**Purpose**: Results from tool execution

**When Emitted**: 
- After tool execution completes
- Contains tool output or error

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "agent": "main",
  "tool_call_id": "call-123",
  "name": "read_file",
  "status": "success" | "error",
  "content": "file contents..." | "error message",
  "error": "error details" (if status is "error")
}
```

**Code Location**: `src/server/handlers/streaming_handler.py:1121`

---

### 5. **`interrupt`**
**Purpose**: Human-in-the-loop (HITL) - plan approval request

**When Emitted**: 
- When agent needs user approval before proceeding
- Typically for plan submission or critical actions

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "interrupt_id": "interrupt-456",
  "action_requests": [
    {
      "description": "Plan: Create new feature...",
      "tool_name": "submit_plan",
      "args": {...}
    }
  ],
  "role": "assistant",
  "finish_reason": "interrupt"
}
```

**Code Location**: `src/server/handlers/streaming_handler.py:635`

---

### 6. **`error`**
**Purpose**: Error messages from workflow execution

**When Emitted**: 
- When workflow encounters an error
- On timeout
- On exceptions

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "error": "Error message here",
  "message": "An error occurred during processing",
  "type": "timeout_error" | "exception" | etc.
}
```

**Code Location**: `src/server/handlers/streaming_handler.py:350, 1232`

---

### 7. **`warning`**
**Purpose**: Warning messages (non-fatal)

**When Emitted**: 
- When workflow is approaching timeout
- Other non-critical warnings

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "message": "Workflow approaching timeout (300s / 600s)",
  "type": "timeout_warning",
  "elapsed_seconds": 300,
  "timeout_seconds": 600
}
```

**Code Location**: `src/server/handlers/streaming_handler.py:332`

---

### 8. **`keepalive`**
**Purpose**: Connection heartbeat to prevent timeout

**When Emitted**: 
- Every 15 seconds during long operations
- Prevents HTTP connection from timing out

**Data Structure**:
```json
{
  "status": "alive"
}
```

**Code Location**: `src/server/handlers/streaming_handler.py:259`

---

### 9. **`subagent_status`**
**Purpose**: Background subagent execution status (legacy)

**When Emitted**: 
- Every 2 seconds when subagents are running
- Shows active and completed subagents

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "active_subagents": ["subagent-1", "subagent-2"],
  "completed_subagents": ["subagent-3"]
}
```

**Note**: This is the legacy format. New code uses `background_task_status`.

**Code Location**: `src/server/handlers/streaming_handler.py:744`

---

### 10. **`background_task_status`**
**Purpose**: Background task execution status (new format)

**When Emitted**: 
- Every 2 seconds when background tasks are running
- Provides richer metadata than `subagent_status`

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "active_tasks": [
    {
      "id": "task-123",
      "description": "Task description",
      "type": "subagent",
      "tool_calls": 5,
      "current_tool": "read_file"
    }
  ],
  "completed_tasks": ["task-456"],
  "timestamp": "2024-01-01T12:00:00Z"
}
```

**Code Location**: `src/server/handlers/streaming_handler.py:725`

---

### 11. **`file_operation`**
**Purpose**: File write/edit operations

**When Emitted**: 
- When agent writes or edits files
- Logs file system changes

**Data Structure**:
```json
{
  "agent": "main",
  "operation": "write_file" | "edit_file",
  "file_path": "/path/to/file.py",
  "tool_call_id": "call-123",
  "timestamp": "2024-01-01T12:00:00Z",
  "status": "success" | "failed",
  "line_count": 50,
  "content": "file contents..." (for write_file),
  "old_string": "..." (for edit_file),
  "new_string": "..." (for edit_file),
  "error": "error message" (if status is "failed")
}
```

**Code Location**: `src/server/handlers/streaming_handler.py:445, 468`

---

### 12. **`summarization_signal`**
**Purpose**: Summarization lifecycle signals

**When Emitted**: 
- When summarization process starts/completes
- For tracking summarization workflow

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "signal": "start" | "complete" | "error",
  "summary_length": 1000 (optional),
  "error": "error message" (if signal is "error")
}
```

**Code Location**: `src/server/handlers/streaming_handler.py:405`

---

### 13. **`credit_usage`**
**Purpose**: Token usage and credit consumption

**When Emitted**: 
- At the end of workflow execution
- Shows aggregated token counts and credits

**Data Structure**:
```json
{
  "thread_id": "abc123",
  "tokens": {
    "input_tokens": 1000,
    "output_tokens": 500,
    "total_tokens": 1500
  },
  "total_credits": 0.15,
  "timestamp": "2024-01-01T12:00:00Z"
}
```

**Note**: Does NOT include USD costs or model names (privacy).

**Code Location**: `src/server/handlers/streaming_handler.py:1297`

## 🔄 Event Flow Example

Typical event sequence for a simple task:

```
1. message_chunk (content: "I'll help you...")
2. tool_calls (name: "read_file", args: {...})
3. tool_call_chunks (chunk: "path")
4. tool_call_result (status: "success", content: "file contents")
5. message_chunk (content: "Here's what I found...")
6. credit_usage (tokens: {...}, total_credits: 0.05)
```

For a plan approval task:

```
1. message_chunk (content: "Let me create a plan...")
2. tool_calls (name: "submit_plan", args: {...})
3. interrupt (interrupt_id: "123", action_requests: [...])
   [User approves/rejects]
4. message_chunk (content: "Proceeding with approved plan...")
5. tool_calls (name: "write_file", args: {...})
6. file_operation (operation: "write_file", status: "success")
7. tool_call_result (status: "success")
8. message_chunk (content: "Done!")
```

## 📊 Event Categories

### **Content Events** (Main output)
- `message_chunk`: Agent responses

### **Tool Events** (Tool execution)
- `tool_calls`: Tool invocation decisions
- `tool_call_chunks`: Streaming tool args (UI only)
- `tool_call_result`: Tool execution results

### **Control Events** (Workflow control)
- `interrupt`: HITL approval requests
- `error`: Error messages
- `warning`: Warning messages

### **Status Events** (Background activity)
- `subagent_status`: Legacy subagent status
- `background_task_status`: New task status format
- `keepalive`: Connection heartbeat

### **Logging Events** (Audit trail)
- `file_operation`: File system changes
- `summarization_signal`: Summarization lifecycle
- `credit_usage`: Token/credit tracking

## 🔍 Event Handling in CLI

**File**: `libs/ptc-cli/ptc_cli/streaming/executor.py`

The CLI handles events in `execute_task()` function (Line 341-430):

```python
if event_type == "message_chunk":
    _handle_message_chunk(event_data, state)
elif event_type == "tool_calls":
    _handle_tool_calls(event_data, state, todo_state)
elif event_type == "tool_call_result":
    _handle_tool_result(event_data, state, tool_name_by_id)
elif event_type == "interrupt":
    # Prompt for plan approval
elif event_type == "error":
    # Display error and break
elif event_type == "done":
    break  # End of stream
elif event_type == "keepalive":
    pass  # Ignore
elif event_type == "subagent_status":
    # Update spinner with subagent count
```

**Note**: The `"done"` event is checked by the CLI but is **not actually emitted by the server**. The stream ends naturally when the workflow completes.

## 💡 Notes

1. **Event Sequence Numbers**: All events have an `id` field for reconnection support
2. **Event Filtering**: CLI filters out subagent output (only shows main agent)
3. **Content Types**: `message_chunk` can have different `content_type` values
4. **Streaming**: Most events are streamed incrementally for real-time feedback
5. **Error Handling**: Errors stop the workflow and break the event loop
